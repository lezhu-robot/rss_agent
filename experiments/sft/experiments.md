## Prompt engineering

实验组内部对比了两种 Prompt 的构成方式：

1. Few-shot（硬编码示例组）: 
   在 Prompt 中带上了 3 条人工精选的覆盖不同领域（产品、传言模型、底层算力）的完整参考样例（Demo）。
   运行命令: `python prompt_mvp/1_run_infer.py`
2. Zero-shot（纯规则组）:
   移除了所有的参考样例，仅保留了字段释义与人工总结的 【分类指导规则】。
   运行命令: `python prompt_mvp/1_run_infer.py --disable-demos`

| 评估指标 | LLaMA3-8B + Few-shot (带 3 个预设示例) | LLaMA3-8B + Zero-shot (仅靠纯规则) | 对比差异 |
| --- | --- | --- | --- |
| MACRO_F1 | 0.3112 | 0.2782 | 0.033 |
| MACRO_PRECISION | 0.3994 | 0.3284 | 0.071 |
| MACRO_RECALL | 0.3223 | 0.2982 | 0.0241 |
| Subject Strict Set Match Rate | 0.46 | 0.42 | 0.04 |
| 全字段完全正确率 (Strict Match) | 0.08 | 0.06 | 0.021 |

从实验数据中，我们可以得出以下结论：

1. LLaMA3-8B 的零样本领域理解能力存在明显短板
   在未经过垂直领域的 SFT 微调时，8B 级别的小模型仅仅依靠系统提示词（Zero-shot），即使在加上严密的 JSON Schema 约束后，其 MACRO_F1 仅约 27.8 分，全六个字段的一致率（Strict Match）仅为 6%。这意味着模型在处理复杂的新闻分类和实体抽取的业务边界时力不从心。
2. Few-shot（少样本提示）对 8B 小模型产生正向增益
   对比两组数据可以看出，给足 3 个 Hardcoded Demos 后，模型的各项预测指标均有明显提升。其中，宏平均 F1 分数上升了 3.3%。对于未经微调、参数规模较小的模型而言，“依葫芦画瓢”的抄答案能力（上下文学习）可以确实弥补其阅读复杂规则并自主推理的不足。
3. 纯提示词工程（Prompt Engineering）达到效能瓶颈
   即使是在加上预设优质例子最优（Few-shot）的前提下，模型最高 F1 仍然只有 31.1%，且实体抽取的精确对齐率未达到及格线（46%）。
   这强有力地验证了本项目后续进阶策略的必要性：面对如此复杂的长尾分类边界情况，我们必须转入 SFT 全量参数微调或 RAG 动态检索增强流程，才有可能实现业务期望的高准确率。

## Prompt + RAG

### 一、 核心探索结论（一句顶一万句）

在这类**“多字段强约束嵌套 + 选项互斥互斥”**的结构化数据抽取任务里，在 8B 小模型的智力基础上： RAG 是伪命题，甚至起反作用。人工精雕细琢的 Prompt（0.346) 远胜千机百变的 FAISS 检索（0.228~0.267)。如果想继续提分，必须抛弃 RAG，全面拥抱 SFT。

---

### 二、 RAG 折腾全过程回顾

我们从“既然纯靠提示词不行，那就给它抄几篇标准答案”的朴素直觉出发，对着 RAG 开展了 4 场大手术：

- Round 1：原始裸跑（MACRO_F1 0.228） 我们直接用 FAISS 搜出 Top-3 喂给 LLaMA3-8B。结果发现分数惨跌，甚至不如什么都不检索的 0.31 基线。
- Round 2：洗稿降噪 + 文本截断（MACRO_F1 0.267） 发现了长文本污染注意力的问题后，我们把 FAISS 返回的 DemoID、DistanceScore 等垃圾信息全剔除，并暴君式地在 150 字处强行截断原文，强行注入最高效的“4条纯手工大字报特征词”。分数成功反弹了将近 4%，但依然摸不到纯 Prompt（0.346）的脚踝。
- Round 3：动态 LLM 课代表蒸馏（MACRO_F1 0.228） 你提出了天才构想：把 RAG 的冗长材料彻底摒弃，让 LLM 做一次预处理，把 3 篇文章压成 3 句“特征词 -> 标签”的规则。想法完美，但败在现实：当总结规则的人和做题的人都是同一个能力见顶的 8B 模型时，左脚踩右脚上不了天，原地踏步。
- Round 4（终局）：多样性感知检索（MACRO_F1 0.246） 为了彻底撕下 RAG 不能提效的遮羞布，我们写脚本诊断拿到了“抢眼的作弊证据”：FAISS 容易找回 3 篇极其雷同、连带错误也雷同的答案（同质化），导致模型 71.4% 都在盲目抄瞎答案。于是我修改了代码，强制找出的 3 篇参考新闻必须分类互斥。结果 F1 略微恢复到 0.246，但依然打不过硬编码。

### 三、 为什么在这条赛道里，RAG 打不过手工硬编码的 Prompt？

经过这四轮交锋，我们把 RAG（基于相似度检索）的 3 个致命死穴彻底摸清了：

1. “信息回音壁”与“视野剥夺” 这是一场分类游戏（例如区别：它是具体应用产品 product 还是底层模型技术 model）。
   - 人写的最优 Prompt 是上帝视角的：“我给你看 3 个例子，第一篇是卖产品的，第二篇是搞底层的，第三篇是炒股票的，你看清楚它们的界限在哪。”——这叫建立判断基准体系。
   - FAISS 的 RAG 是管中窥豹的：“哦你要测‘大模型发布’？太好了，我刚好有 3 篇特别像的‘大模型发布旧闻’全塞给你。”在 8B 模型的眼里，这就变成了一个全选 ABCD 里同选项的考试，一旦 FAISS 把 3 个错误选项聚拢在一起，小模型会立刻陷入认知回音壁，放弃抵抗直接盲从抄写。（我们的分析脚本证明了这带来了 71.4% 的致盲率）。
2. 注意力带宽被长文本挤占 (Context Overflow) 8B 在指令长了之后很容易“神游”。RAG 无论怎么截断，依然强行往上下文里塞进了一堆人名、地名、冗余的描述。相比之下，纯 Prompt 极其干净：几行精炼的中文特征词 + 单刀直入的映射结论。没有噪音，就赢了一半。
3. 知识冲突下的“邯郸学步” 由于新闻本身的多面性，字面上很相似的新闻，其分类落点可能南辕北辙（比如“A 公司并购 B 公司发大模型”，到底是投资并购还是技术大模型）。RAG 生搬硬套的参考答案，极易在这些微小但决定生死的语义转折处，把大模型带进沟里。

### 四、 下一步路在何方？

到这里，外部干预模型（外挂文档、优化提示词、更换检索套路）的潜力已被我们不仅压榨干，还拆解出病理报告了。
想要让模型跨过 0.35 这个在 8B 这个尺度的“常识玻璃顶”，唯有内科手术——上 SFT 技术（Supervised Fine-Tuning）。
通过成百上千条结构化问答对的高质量微调，把这套分类的逻辑钢印、把每一类特征词的敏感度，深深地固化进它几百亿个多维交错的参数权重里去。我们现在随时可以转身推开 SFT 项目的大门了。
